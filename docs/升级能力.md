# true-mem 改造后升级能力

> 本次改造在保持 true-mem v1.4.1 核心插件代码不动的前提下，通过外部脚本 + 行为约定 + 目录基础设施三层叠加，系统性地扩展了记忆体系的深度和广度。

## 二、新增能力详解

### 2.1 时序有效性管理（来源：memory-systems Gotcha 2）

**问题背景：** 事实随时间变化。例如用户从 Python 3.12 升级到 3.13，但 true-mem 同时持有两条记忆，Agent 可能基于旧记忆做决策。

**新增能力：** 通过 `temporal_index.jsonl` 文件建立记忆的"版本谱系"：

```
mem_abc123 (Python 3.12, valid_from: 2026-01-01)
    ↓ 被替代
mem_def456 (Python 3.13, valid_from: 2026-04-15)
```

当注入前执行 `memory-temporal-filter.py`，被标记为 `supersedes` 的旧记忆自动排除，防止上下文毒害。

**使用场景：**
- 技术栈升级
- 项目重命名
- 接口废弃
- 架构决策推翻重做

### 2.2 记忆质量评估体系（来源：memory-systems 基准评估方法）

**四个评估维度：**

```
┌─ 噪声检测 ──────────────────────────────────┐
│ 10 种噪声模式正则匹配                        │
│ · system-reminder 标签                      │
│ · XML 标签（<true_memory_context>等）        │
│ · AI meta-talk（"Goal: The user..."）        │
│ · 问句误判为陈述                             │
│ · 工具指令泄漏                               │
│ · 空内容/纯元数据                            │
│ 评分：1-10 分，≥7 视为噪声                   │
└─────────────────────────────────────────────┘

┌─ 重复检测 ──────────────────────────────────┐
│ 归一化 summary → SHA256 hash               │
│ 检测 O(1) exact 重复                        │
│ 同时发现语义近重复（需人工判断）              │
└─────────────────────────────────────────────┘

┌─ 分类合理性 ────────────────────────────────┐
│ constraint 不含问句                         │
│ preference 不含 AI 元对话                    │
│ procedural 不含工具定义                      │
│ episodic 不含固定偏好                        │
└─────────────────────────────────────────────┘

┌─ 时序交叉验证 ─────────────────────────────┐
│ 加载 temporal_index.jsonl                   │
│ 检查每条的 supersedes 链完整性和合理性        │
│ 标记孤立的替代链（源已被删除）               │
└─────────────────────────────────────────────┘
```

### 2.3 实体关联知识图谱（来源：memory-systems 实体记忆层 + ontology skill）

**能力链：**

```
true-mem (自动记忆提取)
    ↓ memory-to-ontology-bridge.py
ontology (结构化实体+关系)
    ↓ 图查询 / 遍历
下游 Agent (关系感知的上下文检索)
```

**已建立的实体模型：**

| 实体类型 | 来源 | 示例 |
|---------|------|------|
| Person | 固定创建 | SYZ |
| Project | 记忆 project_scope 字段 | amazon-ad |
| Decision | preference / constraint / procedural 分类 | search-mode constraint |
| Language | 关键词匹配（Python, TypeScript, Rust...） | 按需创建 |
| Technology | 关键词匹配（SQLite, Postgres, Docker...） | 按需创建 |

**已建立的关系类型：**

```
Person ──prefers──→ Language / Technology
Person ──constrains──→ Tool / Language
Person ──decided──→ Decision
Project ──has_rule──→ Decision
Project ──uses──→ Technology / Framework
```

### 2.4 上下文窗口保护（来源：filesystem-context 模式1）

**能力：** 当任何工具输出超过 2000 token 时，自动写入 scratch 文件而非注入上下文。

**效果对比：**

```
改造前: Web搜索返回 8000 token → 全部注入上下文 → 占用 ~3000 token 注意力预算
改造后: Web搜索返回 8000 token → 写入 scratch/search_xxx.txt → 返回 100 token 摘要
        后续需要时: grep "关键词" scratch/search_xxx.txt → 精准检索 50 token
```

### 2.5 长任务进度保真（来源：filesystem-context 模式2）

**能力：** 跨 3 轮以上的任务自动创建结构化计划文件，每轮开始时重读恢复方向感。

**效果：**

```
改造前: 5 轮重构任务，第 3 轮后 Agent 遗忘前 2 轮的上下文 → 重复工作或方向偏离
改造后:
  第 1 轮: 创建 plan YAML → steps[1]=completed
  第 2 轮: 读 plan → steps[2]=in_progress
  第 3 轮: 读 plan → steps[2]=completed, steps[3]=in_progress
  ...
  第 N 轮: 从 plan 文件精准恢复状态，0 信息丢失
```

### 2.6 会话启动自检（来源：memory-systems 框架）

**能力：** 每次新会话启动时，Agent 自动执行 §7.4 Session Checklist：

```
1. 检查 .sisyphus/plans/ — 是否有进行中的任务？
2. 检查 <true_memory_context> — 记忆是否与当前代码库一致？
3. 如果矛盾 → 提问确认，而非基于旧记忆操作
4. 工具输出 >2000 token → 自动 scratch
```

## 三、架构升级路径

基于 memory-systems 的渐进式升级路径，改造后的 true-mem 现状和未来方向：

```
当前状态（改造后）:
  ✅ 文件系统 LTM（SQLite）
  ✅ 时序有效性管理
  ✅ 质量评估探针
  ✅ 实体关联（简单图）
  ✅ Scratch Pad + Plan Persistence

升级信号监控（按需评估）:
  □ 语义检索需求出现（"Agent 找不到我已经告诉过它的事"）
  □ 多跳推理需求（"关于 X 的所有相关决定"需要跨记忆关联）
  □ 记忆量级突破阈值（活跃记忆 > 200 条）
  □ 跨项目知识迁移需求

对应升级路径:
  信号 1 → 开启 TRUE_MEM_EMBEDDINGS=1 → 实现 query-aware 重排序
  信号 2 → 引入 ontology 图遍历 → 实现关系链查询
  信号 3 → 评估 Zep/Graphiti 时序 KG → 实现复杂关系推理
  信号 4 → 评估 Cognee 多层语义图 → 实现跨领域自适应记忆
```

## 四、运维能力

### 4.1 日常巡检

```bash
# 会话启动检查
py scripts/session-check.py

# 质量审计
py scripts/memory-eval.py

# 清理 scratch
.\scripts\scratch-cleanup.ps1 -DryRun  # 先预览
.\scripts\scratch-cleanup.ps1           # 执行
```

### 4.2 手动时序标记

```bash
# 标记"前端从 Vue 迁移到 React"
echo '{"memory_id":"mem_xxx","subject":"前端框架","valid_from":"2026-04-30","supersedes":"mem_yyy"}' \
  >> ~/.config/opencode/true-mem/temporal_index.jsonl

# 验证过滤效果
py scripts/memory-temporal-filter.py
```

### 4.3 实体图谱维护

```bash
# 增量同步
py scripts/memory-to-ontology-bridge.py --apply

# 查看图谱
Get-Content memory/ontology/graph.jsonl -Tail 10
```

## 五、兼容性保证

| 保证项 | 说明 |
|--------|------|
| **不修改 true-mem 源** | 所有脚本在外部独立运行，升级 true-mem npm 包不受影响 |
| **DB 只读查询** | memory-eval.py 和 bridge.py 只 SELECT，不修改 true-mem 原始数据 |
| **删除可恢复** | `--fix` 将 status 标记为 deleted，不物理删除行 |
| **JSONL append-only** | ontology graph.jsonl 和 temporal_index.jsonl 均为追加模式，不发散 |
| **回滚简单** | 删除 `scripts/` 下对应脚本即可回退，不影响 true-mem 核心功能 |
